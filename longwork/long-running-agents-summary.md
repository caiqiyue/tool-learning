# 长时间运行 Agent 框架：通俗理解版

> 这是一篇关于如何让 AI Agent 长时间稳定工作的文章核心内容的精简总结。

---

## 核心问题：Agent 的"失忆症"

想象你是一家软件公司的工程师，每天早上上班时发现：
- 昨天的代码不知道是谁写的
- 没人告诉你项目做到哪了
- 你甚至不知道这个项目是做什么的

这就是 AI Agent 面临的问题——每个新的"会话"就像一个新工程师入职，什么都不记得。

## 两步走解决方案

### 第一步：初始化 Agent（"开国功臣"）

**它的任务是打地基**，专门在第一次运行时做：

1. **写一个功能清单**（`feature_list.json`）
   - 把大任务拆成 200+ 个小功能
   - 每个功能写清楚：要做什么、怎么做、现在通过没有
   - 一开始全标记为"未通过"

2. **写一个进度记录**（`claude-progress.txt`）
   - 像工作日志一样，记录"今天做了什么"
   - 让下一个 Agent 能快速了解状态

3. **写一个启动脚本**（`init.sh`）
   - 一键启动开发服务器
   - 不用每次都手动配置环境

4. **做第一次 Git 提交**
   - 把初始代码库固定下来
   - 方便以后回溯

### 第二步：编码 Agent（"每天进步的工人"）

**它的任务是增量工作**，每次会话都：

1. **先"看病"**：看看项目现在啥状态
   - 运行 `pwd` 确认目录
   - 读进度记录了解上次的进度
   - 看 Git 日志了解历史
   - 跑一下基本功能测试，确认没把项目搞坏

2. **只做一件事**：从功能清单里挑一个没完成的
   - 一次只做一个功能，不会贪多嚼不烂
   - 做好了就标记为"通过"

3. **收尾工作**：让环境保持干净
   - Git 提交代码（带清晰说明）
   - 更新进度日志

## 四大失败模式及应对

### 1. Agent 过早宣布"完成了"

- **症状**：做了 30% 就觉得大功告成
- **药方**：功能清单。每个功能都要单独验证。

### 2. 代码留下 bug 或混乱

- **症状**：下次来发现代码跑不通了
- **药方**：每次结束前跑测试 + Git 提交 + 写日志

### 3. 功能没测完就标记"完成"

- **症状**：以为好了，其实没好
- **药方**：强制要求端到端测试（模拟真实用户操作）

### 4. 每次都要花时间配置环境

- **症状**：花 30 分钟搭建环境，真正工作没做多少
- **药方**：初始化阶段写好 `init.sh`，一键启动

## 我的思考

### 为什么这个框架有效？

1. **解决了"交接"问题**
   - 传统开发中，工程师交接需要大量文档
   - 这个框架强制每次交接都有结构化的记录

2. **强制"小步快跑"**
   - 一次只做一个功能，减少了出错面积
   - 错了也好定位，好回滚

3. **引入了"质量门禁"**
   - 不测试不标记通过
   - 类似 CI/CD 的理念

### 启发

这个框架本质上是把**人类的工程实践**自动化了：
- 功能清单 → 任务分解
- Git 提交 → 版本控制
- 进度日志 → 团队沟通
- 端到端测试 → 质量保证

Agent 不是神奇的万能药，它需要**像管理人类团队一样管理它**。

### 还能怎么改进？

我觉得未来可以加入：
- **专门的测试 Agent**：专注找 bug
- **代码审查 Agent**：专注提升代码质量
- **文档 Agent**：专注维护文档

就像一个团队里有前端、后端、测试，Agent 也可以有分工。
