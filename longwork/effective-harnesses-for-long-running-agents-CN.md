# 长时间运行 Agent 的有效框架

发布时间：2025年11月26日

Agent 在跨多个上下文窗口工作时仍面临挑战。我们向人类工程师寻求灵感，创建了一个更有效的长时间运行 Agent 框架。

随着 AI Agent 变得越来越强大，开发者越来越多地让它们承担复杂的任务，这些任务需要耗时数小时甚至数天的工作。然而，让 Agent 在多个上下文窗口之间保持一致的进展仍是一个尚未解决的问题。

长时间运行 Agent 的核心挑战在于它们必须以离散的会话方式工作，而每个新会话开始时都没有之前的记忆。想象一下一个软件项目由轮班的工程师管理，每个新工程师到达时都不记得前一班发生了什么。因为上下文窗口是有限的，而且大多数复杂项目无法在单个窗口内完成，所以 Agent 需要一种方法来弥合编码会话之间的间隙。

我们开发了一个双管齐下的解决方案，使 [Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview) 能够跨多个上下文窗口有效工作：一个**初始化 Agent**，在首次运行时设置环境；以及一个**编码 Agent**，负责在每个会话中取得增量进展，同时为下一个会话留下清晰的产物。你可以在配套的 [快速入门](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding) 中找到代码示例。

## 长时间运行 Agent 的问题

Claude Agent SDK 是一个强大的通用 Agent 框架，擅长编码以及其他需要模型使用工具收集上下文、制定计划和执行的任务。它具有上下文管理能力，如压缩功能，使 Agent 能够在不耗尽上下文窗口的情况下处理任务。理论上给定这种设置，Agent 应该能够持续进行任意长时间的有用工作。

然而，压缩是不够的。开箱即用，即使是在 Claude Agent SDK 中循环运行的前沿编码模型（如 Opus 4.5），如果只给出一个高级提示（如"构建一个 [claude.ai](http://claude.ai/redirect/website.v1.7a18f2a6-dc54-4330-8d00-82ccfc6696be) 的克隆"），也难以构建一个生产质量的 Web 应用。

Claude 的失败表现为两种模式。首先，Agent 倾向于一次做太多事情——本质上是试图一次性完成整个应用。通常，这会导致模型在实现过程中耗尽上下文，导致下一个会话从一个半实现且未记录的功能开始。然后 Agent 不得不猜测发生了什么，并花费大量时间试图让基本应用重新工作。即使有压缩，这种情况也会发生，因为压缩并不总是能完美地将指令传递给下一个 Agent。

第二种失败模式通常发生在项目的后期。在一些功能已经构建之后，后来的 Agent 实例会四处看看，看到已经取得了进展，然后就宣布工作完成。

这将问题分解为两个部分。首先，我们需要设置一个初始环境，为给定提示所需的所有功能奠定基础，这为 Agent 逐步和按功能工作奠定了基础。其次，我们应该提示每个 Agent 在向目标取得增量进展的同时，在会话结束时将环境保持在干净的状态。所谓"干净状态"，我们指的是适合合并到主分支的代码：没有重大 bug，代码有序且有良好的文档，通常开发者可以轻松地开始新功能的工作，而不必首先清理一个不相关的烂摊子。

在内部实验时，我们使用双部分解决方案解决了这些问题：

1. 初始化 Agent：第一个 Agent 会话使用专门的提示，要求模型设置初始环境：一个 `init.sh` 脚本、一个记录 Agent 所做工作的 `claude-progress.txt` 文件，以及一个显示添加了哪些文件的初始 git 提交。
2. 编码 Agent：每个后续会话都要求模型取得增量进展，然后留下结构化的更新。

关键洞察是找到一种方法让 Agent 在以全新上下文窗口启动时能够快速理解工作状态，这通过 `claude-progress.txt` 文件和 git 历史来实现。这些实践的灵感来自对有效软件工程师每天做什么的了解。

## 环境管理

在更新的 [Claude 4 提示工程指南](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices#multi-context-window-workflows) 中，我们分享了多上下文窗口工作流程的一些最佳实践，包括使用"第一个上下文窗口使用不同提示"的框架结构。这种"不同的提示"要求初始化 Agent 设置所有必要上下文的环境，供未来的编码 Agent 有效工作。在这里，我们深入探讨这种环境的一些关键组件。

### 功能列表

为了解决 Agent 一次性完成应用或过早认为项目已完成的問題，我们提示初始化 Agent 编写一个全面的功能需求文件，扩展用户最初的提示。在 [claude.ai](http://claude.ai/redirect/website.v1.7a18f2a6-dc54-4330-8d00-82ccfc6696be) 克隆示例中，这意味着超过 200 个功能，例如"用户可以打开新聊天、输入查询、按回车键并看到 AI 响应"。这些功能最初都标记为"未通过"，以便后来的编码 Agent 有一个清晰的功能完整轮廓。

```json
{
    "category": "functional",
    "description": "新建聊天按钮创建一个新的对话",
    "steps": [
      "导航到主界面",
      "点击'新建聊天'按钮",
      "验证新对话已创建",
      "检查聊天区域显示欢迎状态",
      "验证对话出现在侧边栏"
    ],
    "passes": false
  }
```

我们提示编码 Agent 只通过更改 `passes` 字段的状态来编辑此文件，并且我们使用强有力的指示，如"删除或编辑测试是不可接受的，因为这可能导致功能缺失或 bug"。经过一些实验，我们最终使用 JSON 格式，因为与 Markdown 文件相比，模型不太可能不当更改或覆盖 JSON 文件。

### 增量进展

有了这个初始环境脚手架，编码 Agent 的下一次迭代被要求一次只处理一个功能。这种增量方法被证明是解决 Agent 倾向于一次做太多事情的问题的关键。

在逐步工作之后，模型在做出代码更改后仍然必须将环境保持在干净的状态。在我们的实验中，我们发现引出这种行为的最佳方式是要求模型用描述性的提交信息将进度提交到 git，并在进度文件中写入进度摘要。这允许模型使用 git 来还原错误的代码更改并恢复代码库的工作状态。

这些方法也提高了效率，因为它们消除了 Agent 必须猜测发生了什么并花费时间试图让基本应用重新工作的需要。

### 测试

我们观察到的最后一个主要失败模式是 Claude 倾向于在未经适当测试的情况下将功能标记为完成。在没有明确提示的情况下，Claude 倾向于做出代码更改，甚至使用单元测试或 `curl` 命令对开发服务器进行测试，但无法认识到功能没有端到端工作。

在构建 Web 应用的情况下，一旦明确提示使用浏览器自动化工具并像人类用户一样进行所有测试，Claude 在验证功能端到端方面做得很好。

为 Claude 提供这些类型的测试工具显著提高了性能，因为 Agent 能够识别和修复仅从代码中不明显的 bug。

一些问题仍然存在，比如 Claude 的视觉能力和浏览器自动化工具的局限性使得识别每种类型的 bug 都很困难。例如，Claude 无法通过 Puppeteer MCP 看到浏览器原生的警报弹窗，依赖这些弹窗的功能往往会有更多 bug。

## 快速上手

有了以上所有内容，每个编码 Agent 都被提示运行一系列步骤来获取方向，有些很基本但仍然有帮助：

1. *运行 `pwd` 查看你正在工作的目录。你只能编辑这个目录中的文件。*
2. *阅读 git 日志和进度文件，了解最近做了什么工作。*
3. *阅读功能列表文件，选择尚未完成的最高优先级功能开始工作。*

这种方法在每个会话中为 Claude 节省了一些 tokens，因为它不需要弄清楚如何测试代码。它还有助于要求初始化 Agent 编写一个可以运行开发服务器的 `init.sh` 脚本，然后在实现新功能之前运行基本的端到端测试。

在 claude.ai 克隆的情况下，这意味着 Agent 总是启动本地开发服务器，并使用 Puppeteer MCP 开始新聊天、发送消息并接收响应。这确保了 Claude 能够快速识别应用是否处于损坏状态，并立即修复任何现有的 bug。如果 Agent 反而开始实现新功能，可能会使问题变得更糟。

考虑到所有这些，典型会话开始时会有以下助手消息：

```
[助手] 我将从获取方向并理解项目的当前状态开始。
[工具使用] <bash - pwd>
[工具使用] <读取 - claude-progress.txt>
[工具使用] <读取 - feature_list.json>
[助手] 让我检查 git 日志以了解最近的工作。
[工具使用] <bash - git log --oneline -20>
[助手] 现在让我检查是否有 init.sh 脚本可以重启服务器。
<启动开发服务器>
[助手] 太好了！现在让我导航到应用程序并验证一些基本功能是否仍然正常工作。
<测试基本功能>
[助手] 根据我的验证测试，我可以看到基本功能运行良好。核心聊天功能、主题切换、对话加载和错误处理都正常工作。现在让我更全面地阅读 tests.json 文件，了解接下来需要实现什么。
<开始新功能的工作>
```

## Agent 失败模式及解决方案

| **问题** | **初始化 Agent 行为** | **编码 Agent 行为** |
| --- | --- | --- |
| Claude 太早宣布整个项目胜利。 | 设置功能列表文件：基于输入规范，设置一个结构化的 JSON 文件，包含端到端功能描述列表。 | 在会话开始时阅读功能列表文件。选择一个单一功能开始工作。 |
| Claude 留下的环境有 bug 或未记录进度。 | 编写初始 git 仓库和进度笔记文件。 | 通过阅读进度笔记文件和 git 提交日志开始会话，并在开发服务器上运行基本测试以捕获任何未记录的 bug。结束时编写 git 提交和进度更新。 |
| Claude 过早将功能标记为完成。 | 设置功能列表文件。 | 自我验证所有功能。只在仔细测试后将功能标记为"通过"。 |
| Claude 必须花时间弄清楚如何运行应用。 | 编写可以运行开发服务器的 `init.sh` 脚本。 | 通过阅读 `init.sh` 开始会话。 |

长时间运行 AI Agent 中四种常见失败模式及解决方案。

## 未来工作

这项研究展示了一组可能的解决方案，用于构建长时间运行的 Agent 框架，使模型能够在多个上下文窗口中取得增量进展。然而，仍有一些开放性问题。

最值得注意的是，目前尚不清楚单一通用编码 Agent 是否在不同上下文中表现最佳，或者是否可以通过多 Agent 架构实现更好的性能。似乎有理由认为，像测试 Agent、质量保证 Agent 或代码清理 Agent 这样的专门 Agent，可能会在软件开发生命周期中的子任务上做得更好。

此外，这个演示针对全栈 Web 应用开发进行了优化。一个未来方向是将这些发现推广到其他领域。这些经验教训中的一些或全部很可能可以应用于例如科学研究或金融建模所需的其他类型的长时间运行 Agent 任务。
