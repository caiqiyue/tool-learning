# 第一节课：Docker 入门认识

## 1. 什么是 Docker

### 专业解释

Docker 是一个用于构建、分发、运行应用的容器化平台。它可以把应用程序及其运行时依赖，例如代码、系统工具、库、配置文件，打包成一个标准化的运行单元，这个运行单元通常叫做**容器**。

Docker 的核心目标不是“创建一台虚拟机”，而是让应用在不同环境中都能以更一致的方式运行。开发者可以在本地把应用打包成镜像，然后在测试环境、生产环境、云服务器中以几乎相同的方式启动它。

### 通俗理解

你可以把 Docker 理解成一个“标准化打包盒 + 运行盒”。

- 以前你写好一个项目，拿到另一台机器上，经常会遇到：
  - Python 版本不对
  - Java 依赖不全
  - 数据库驱动没装
  - 系统库版本不一致
- Docker 的做法是：
  - 先把程序和它要用的环境一起装进盒子里
  - 再把这个盒子拿到哪里都按同样方式运行

所以 Docker 的核心价值可以用一句话概括：

**把“应用运行环境”标准化，让软件在任何地方都尽量做到一致运行。**

## 2. 为什么会出现 Docker

### 2.1 Docker 出现之前的典型问题

在 Docker 流行之前，软件部署常见的痛点非常多。

#### 问题一：开发环境和生产环境不一致

开发同学在自己电脑上跑得好好的，部署到服务器后却报错。这类问题常被概括为：

**“在我电脑上是好的。”**

原因通常包括：

- 操作系统不同
- 依赖库版本不同
- 环境变量不同
- 安装路径不同
- 系统组件缺失

#### 问题二：应用之间相互污染

一台服务器上如果运行多个项目，经常会出现依赖冲突。

例如：

- A 项目需要 `Python 3.8`
- B 项目需要 `Python 3.11`
- C 项目依赖某个旧版本的 OpenSSL

这些程序装在同一台机器上，就容易互相影响，维护起来很痛苦。

#### 问题三：部署效率低

传统部署往往要手动做很多事情：

- 安装运行环境
- 安装依赖
- 修改配置
- 启动服务
- 检查端口
- 排查权限问题

一旦环境多了，重复劳动就会非常明显。

#### 问题四：资源利用率不高

在 Docker 更流行之前，很多团队会使用虚拟机来做隔离。虚拟机虽然能隔离环境，但它比较重：

- 每个虚拟机都要带一个完整操作系统
- 启动慢
- 占用内存和磁盘更多
- 同一台物理机能跑的实例数量有限

这就需要一种比虚拟机更轻量的隔离方案。

### 2.2 Docker 为什么能流行

Docker 之所以快速普及，是因为它同时满足了几个关键诉求：

- 让应用和环境一起交付
- 提高部署一致性
- 提供轻量级隔离
- 启动速度快
- 资源利用率高
- 非常适合微服务和云原生场景

### 通俗理解

Docker 的出现，本质上是为了回答一个老问题：

**“怎么才能让程序在别人电脑上、测试环境、服务器上都稳定跑起来？”**

它不是单纯为了“炫技”，而是为了解决部署困难、环境混乱、效率低下这些很现实的问题。

## 3. Docker 的功能是什么

Docker 的功能可以从“打包、运输、运行、隔离、管理”这几个角度理解。

### 3.1 标准化打包

Docker 可以把应用和依赖一起打成一个**镜像（Image）**。

镜像可以理解为一个只读模板，它里面包含：

- 应用代码
- 依赖库
- 运行时环境
- 默认启动命令
- 必要配置

有了镜像，就相当于把应用做成了一个可复制的标准品。

### 3.2 标准化运行

镜像启动后会变成**容器（Container）**。容器就是镜像的运行实例。

这意味着：

- 同一个镜像可以启动多个容器
- 每个容器运行方式一致
- 应用部署动作可以被标准化

### 3.3 环境隔离

容器之间彼此隔离，不同应用可以拥有各自的文件系统视图、进程空间、网络环境和资源限制。

这可以有效减少：

- 依赖冲突
- 应用互相影响
- 运维管理混乱

### 3.4 快速交付和迁移

Docker 镜像可以在开发机、测试机、服务器、云平台之间分发。

所以它非常适合：

- 本地开发
- 持续集成与持续部署
- 灰度发布
- 快速扩缩容
- 故障恢复

### 3.5 版本管理和复现

镜像是可以版本化的，例如：

- `nginx:1.25`
- `redis:7.2`

这使得你可以明确知道当前运行的是哪个版本，也更容易回滚和复现问题。

### 通俗理解

Docker 的功能，简单说就是：

- 帮你把程序装箱
- 帮你把箱子运输到别的机器
- 帮你把箱子快速启动起来
- 让不同程序各住各的“小房间”

## 4. Docker 主要解决了什么问题

这是学习 Docker 时最关键的一部分。你不要只记命令，更要知道它在解决什么问题。

### 4.1 解决“环境不一致”问题

这是 Docker 最经典的价值。

开发环境、测试环境、生产环境往往不一样。Docker 通过镜像把环境一起打包，从而尽量保证运行一致性。

### 4.2 解决“依赖冲突”问题

每个项目都可以运行在自己的容器里：

- 你用你的依赖版本
- 我用我的依赖版本
- 大家互不干扰

### 4.3 解决“部署复杂”问题

以前部署一个项目，可能要写很多安装文档。现在只要：

1. 拉取镜像
2. 启动容器

部署流程会简单很多，也更容易自动化。

### 4.4 解决“迁移困难”问题

应用从一台机器迁移到另一台机器时，以前常常要重装环境。使用 Docker 后，只要目标主机有 Docker，就可以直接运行相同镜像。

### 4.5 解决“隔离不彻底”问题

同一台主机可以同时运行多个容器，它们互相隔离，比传统直接在主机上裸跑多个应用更清晰、更安全、更好管理。

### 4.6 解决“扩容效率低”问题

容器启动很快，所以在业务增长时，可以更容易地横向扩容。例如某个服务压力大了，就多启动几个同样的容器实例。

### 一句话总结

Docker 主要解决的是：

**应用交付过程中的环境一致性、部署效率、隔离性、迁移性和可维护性问题。**

## 5. Docker 的大概底层技术是什么

这一部分我会尽量讲专业，但也尽量让你能听懂。

### 5.1 Docker 并不是虚拟机

很多初学者容易误以为 Docker 就是“轻量版虚拟机”，这不准确。

虚拟机的特点是：

- 通过 Hypervisor 虚拟出完整硬件
- 每个虚拟机都有自己的完整操作系统内核和用户空间

而 Docker 容器通常是：

- 共享宿主机内核
- 只隔离用户空间和运行环境

所以容器更轻、更快，资源占用更少。

### 5.2 Docker 依赖 Linux 容器技术

Docker 的底层核心并不是凭空创造出来的，它主要是对 Linux 已有能力进行了工程化封装。

其中最重要的两类技术是：

- **Namespaces**
- **Cgroups**

#### 1. Namespaces：做“隔离”

Namespaces 用来让不同进程“看到的世界不一样”。

它可以隔离：

- 进程编号
- 网络设备和网络栈
- 挂载点
- 主机名
- 进程间通信资源
- 用户和用户组

比如一个容器里的进程会觉得自己像是在单独的一台机器里运行，它看到的进程列表、网络接口、文件系统挂载情况，不一定和宿主机一致。

通俗理解：

**Namespaces 就像给每个容器戴上一副“独立世界”的眼镜。**

#### 2. Cgroups：做“资源控制”

Cgroups 的作用是限制、统计、隔离进程组对系统资源的使用。

它可以控制的资源包括：

- CPU
- 内存
- 磁盘 I/O
- 网络等资源

例如可以规定：

- 某个容器最多使用多少内存
- 某个容器最多占用多少 CPU

通俗理解：

**Cgroups 就像给每个容器设定“饭量”和“活动范围”。**

### 5.3 Union File System：分层文件系统

Docker 镜像并不是一个简单的大压缩包，它通常采用分层存储。

这背后依赖的是联合文件系统思想，例如 OverlayFS。

它的优势是：

- 多个镜像可以共享底层层
- 节省磁盘空间
- 构建镜像更高效
- 修改时只新增变化层

例如：

- 基础镜像是一层
- 安装 Nginx 是一层
- 拷贝你的项目代码又是一层

这样镜像管理会更灵活。

通俗理解：

Docker 镜像像“乐高积木”一样，一层层叠起来，而不是每次都从零造一遍。

### 5.4 Container Runtime：容器运行时

Docker 并不是直接自己完成所有底层动作，它背后还涉及容器运行时。

典型链路可以粗略理解为：

- Docker CLI：你在命令行输入 `docker run`
- Docker Engine：负责解析和协调
- containerd：负责更底层的容器管理
- runc：真正按 OCI 规范创建和启动容器

这说明 Docker 更像是一个完整的容器平台，而不只是一个简单命令工具。

### 5.5 镜像仓库机制

Docker 生态里还有镜像仓库，例如 Docker Hub。

它的作用类似“软件包仓库”：

- 存镜像
- 拉镜像
- 分发镜像
- 管理版本

这让 Docker 不只是本地运行工具，而是具备完整交付链路。

## 6. 用专业语言重新总结一遍

从专业角度看，Docker 是一个基于 Linux 内核隔离与资源控制能力构建的容器化平台。它通过镜像分层、容器运行时、镜像仓库和标准化接口，把应用打包、分发、部署、运行这整条链路统一了起来。

它的核心价值包括：

- 统一应用交付格式
- 提升环境一致性
- 提供轻量级进程隔离
- 提高资源利用率
- 支持快速部署与弹性扩缩容

Docker 不是传统虚拟机替代品的简单缩小版，而是一套围绕“应用交付与运行一致性”构建的工程体系。

## 7. 用大白话再总结一遍

如果你现在还不想记太多专业词，可以先记住下面这几句话：

- Docker 是一个装程序的标准盒子
- 盒子里不只装代码，还装运行环境
- 程序换台机器也能更稳定地跑起来
- 不同程序可以分别装进不同盒子里，互不干扰
- 它底层靠的是 Linux 的隔离和资源控制能力
- 它比虚拟机更轻，所以启动更快、更省资源

## 8. 初学者这一节最该记住的知识点

### 必记概念

- Docker 是容器平台，不是传统虚拟机
- 镜像是模板，容器是运行中的实例
- Docker 的核心价值是标准化交付和环境一致性
- Docker 主要解决部署、迁移、隔离、依赖冲突等问题
- 底层核心技术是 Namespaces、Cgroups、分层文件系统和容器运行时

### 容易混淆的点

#### Docker 和虚拟机的区别

- 虚拟机：虚拟的是整台机器，通常带完整操作系统
- Docker：更偏向进程级隔离，共享宿主机内核

#### 镜像和容器的区别

- 镜像：静态模板，类似“安装包”或“蓝图”
- 容器：镜像启动后的运行实例，类似“正在运行的程序”
